# AI Usage

This document is a candid account of how AI tools (Claude Code) were used during development of
the AnsrMart grocery checkout system, what was generated vs hand-authored, and what was independently
verified.

---

## How AI was used

### Scaffolding & boilerplate
AI was used to generate the initial monorepo structure (package.json files, workspace wiring,
ESLint/Prettier/Vitest configs), the Supabase SQL migrations, and the Dockerfile/docker-compose.

### Domain layer (`packages/domain`)
The domain logic was the most carefully supervised section. AI produced the initial drafts of:
- `money.js` — all arithmetic helpers (fromRupees, mulQuantity, percentage, halfUp)
- `cart.js` — O(n) resolveCartContext with Map index
- `offers/` — the OfferRule base class and all four strategy implementations
- `tax/engine.js` — pro-rata discount allocation using the largest-remainder method
- `billing/computeBill.js` — the pipeline facade

**What was hand-verified in every case:**
- The rounding mode (`halfUp = floor(n + 0.5)`) — confirmed correct vs JavaScript's `Math.round`
  which uses "round half to even" for .5 cases
- The stacking/precedence policy (3-phase engine A→B→C) — traced manually through worked examples
- The pro-rata tax allocation — verified that integers always sum to `taxableAmount` (invariant test)
- The BuyXGetYFree cheapest-unit selection — traced the sort + slice logic by hand
- `clampToZero` behavior — confirmed `sub(a, b)` never returns negative

### Test suite (87 domain + 17 API = 104 tests)
All tests were generated by AI then inspected line-by-line. Key manual checks:
- The 500-item linearity test — confirmed the timing assertion is on elapsed ms, not a mock clock
- The multi-rate tax breakdown test — re-calculated the expected values manually (₹40+₹250+₹40
  cart, GST 0%/12%/18%, confirmed `taxableAmount = 33000`)
- The worked-example `computeBill` tests — traced the full pipeline (cart → offers → tax → bill)
  and confirmed each assertion against the CONTRACT.md invariants

### API layer (`apps/api`)
AI generated the repository adapters, service orchestration, Zod schemas, and Fastify routing.
The lazy-import pattern for supabase-js (to prevent WebSocket constructor firing in test
environments) was identified and implemented after a test failure.

### Frontend (`apps/web`)
AI generated all React components, the Tailwind design-system config, and the `useCart` hook.
The Botanical/Organic design system tokens (colors, typography, shadows) were provided by the
human and implemented by AI.

---

## What was NOT AI-generated

- `CONTRACT.md` — the domain contract (shapes, invariants, stacking policy) was human-authored
  first and used as the spec for AI code generation
- `plan.md` / `task.md` — architecture decisions, module boundaries, and design-pattern choices
- The GST rate validation constraint (`[0, 500, 1200, 1800]`) — intentionally illustrative, not
  authoritative tax law. Documented explicitly in both CONTRACT.md and the README.
- The "cheapest units free" interpretation of BuyXGetYFree — a deliberate product decision,
  not obvious from a spec; hand-specified and verified in the test suite

## Trust boundaries

| Area | Trust level | Verification method |
|---|---|---|
| Integer paise arithmetic | High — pure math | 22 unit tests; manual trace of rounding cases |
| Stacking/precedence policy | High | 28 offer tests; worked-example integration tests |
| Pro-rata tax allocation | High | 10 tax tests; invariant assertion (sum check) |
| Supabase RLS policies | Medium | Read-through; would need a running Supabase to integration-test |
| Docker build | Low — not tested in CI | Syntax-checked; needs a real build to verify layer caching |
| Admin auth (JWT check) | Medium | Unit-tested middleware; Supabase JWT verification not mocked |
